That brings about an opportunity to use more basic concepts and merge them. Ideally after identifying all the pieces that need to be solved, I would go to pseudocode if the solution to the initial problem statement is a simple algorithm otherwise, I'd go with Design Patterns and see if any of the defined design patterns have modelled the solution beforeand if they have, I would then adapt the pattern to solve the current problem. When I can't match a design pattern to the problem I then go to UML diagrams to see if there could be a potential solution. This is not necessarily a strict order, sometimes given other constraints, pseudocode might need to be done after finding the design pattern or any other combination of the above-mentioned options... or any other step to solve the problem - I am a problem solver.
